## File Relationships

The dependency management follows this hierarchy:

1. **pyproject.toml** is the source of truth
   - Manually edit this file to add/remove dependencies (or use `poetry add/remove`)
   - Contains human-readable version constraints

2. **poetry.lock** is generated from pyproject.toml
   - Auto-generated by Poetry when dependencies change
   - Contains exact pinned versions of all packages and their dependencies
   - Never edit manually

3. **No requirements.txt** in this project
   - Could be generated with `poetry export -f requirements.txt` if needed for pip compatibility
   - Not currently needed as all developers use Poetry

## Update Strategy

- **Which files to update manually**:
  - pyproject.toml (only the `[tool.poetry]` metadata section)
  - Tool configurations ([tool.black], [tool.mypy], etc.)

- **Which files are auto-updated**:
  - poetry.lock (updated by Poetry commands)
  - pyproject.toml dependencies (updated by `poetry add/remove/update`)

- **Update Order**:
  1. Run `poetry add/remove/update` command
  2. Poetry updates pyproject.toml dependencies
  3. Poetry regenerates poetry.lock
  4. Commit both files to git

## Notes

This project uses Poetry's recommended single-file configuration approach. All dependency and tool configuration is centralized in pyproject.toml, following PEP 518 standards. The lock file is committed to ensure all team members and CI/CD use identical dependency versions.

If pip compatibility is ever needed, requirements.txt can be generated with:
```bash
poetry export -f requirements.txt --output requirements.txt
```
